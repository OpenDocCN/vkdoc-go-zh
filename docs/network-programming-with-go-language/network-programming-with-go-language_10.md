# 失效点

分布式应用程序运行在复杂环境中，这使得它们比单台计算机上的独立应用程序更容易发生故障。失效点包括以下内容：

- **客户端错误**
  - 应用程序的客户端可能崩溃（内存不足、除以零）。
  - 客户端系统可能出现硬件问题（绊到电源线）。
  - 客户端的网卡可能失效。

- **网络错误**
  - 网络争用可能导致超时（服务器响应缓慢，包括路由器）。
  - 可能存在网络地址冲突（路由到错误的主机，即使能找到主机）。
  - 路由器等网络组件可能失效。
  - 传输错误可能导致消息丢失。
  - DNS 配置错误。

- **客户端-服务器错误**
  - 客户端和服务器版本可能不兼容（API 变更——HTTP 路径、默认端口配置等）。

- **服务器错误**
  - 服务器系统可能出现硬件问题（包括网卡）。
  - 服务器软件可能崩溃（内存不足、除以零）。
  - 服务器的数据库可能损坏（去重过程中途失败）。

应用程序的设计必须考虑到这些可能的失效。如果系统其他部分发生故障，一个组件执行的任何操作都必须能够恢复。需要采用事务和持续错误检查等技术来避免错误。值得注意的是，虽然独立应用程序对可能发生的错误有很大的控制权，但分布式系统并非如此。例如，服务器无法控制网络或客户端错误，只能准备好处理这些错误。在许多情况下，错误的原因可能无法得知：是客户端崩溃了还是网络中断了？通常，最复杂的开发工作往往是为了拥有一个稳定且可预测的分布式系统。
