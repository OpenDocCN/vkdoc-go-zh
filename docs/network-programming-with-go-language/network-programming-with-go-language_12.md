# 关于分布式计算的思考

以下是构建分布式系统（无论是软件、硬件还是两者兼有）时的考虑因素。

## 透明性

透明性的概念意味着细节对我们隐藏。为这些概念实现解决方案并非易事，但仍是一个合理的设计目标。分布式系统的“圣杯”是提供以下特性：

- 访问透明性
- 位置透明性
- 迁移透明性
- 复制透明性
- 并发透明性
- 可扩展性透明性
- 性能透明性
- 故障透明性

### 访问透明性

用户不应该知道（或需要知道）对系统全部或部分的访问是本地的还是远程的。

### 位置透明性

服务的位置应该无关紧要。

### 迁移透明性

如果系统的一部分迁移到另一个位置，对用户来说应该毫无区别。

### 复制透明性

系统运行一个副本还是多个副本应该无关紧要。

### 并发透明性

系统并发运行的部分之间不应存在干扰。例如，如果我正在访问数据库，那么你不应该知道这件事。

### 可扩展性透明性

系统上有一个用户还是一百万个用户应该无关紧要。

### 性能透明性

性能不应受到任何系统或网络特性的影响。

### 故障透明性

系统不应该失败。如果部分系统失效，系统应在用户不知晓故障发生的情况下恢复。

大多数这些透明性因素更多的是在违背中看到，而非遵守。也有一些显著案例几乎达到了这些要求。例如，当你连接到谷歌时，你不知道（也不关心）服务器在哪里。使用亚马逊云服务的系统能够根据需求进行扩展或缩减。网飞公司拥有近乎残忍的测试策略，定期且故意破坏其系统的大部分组件，以确保整个系统仍然正常工作。

## 分布式计算的八个谬误

太阳微系统公司是一家在分布式系统领域做了大量早期工作的公司，甚至有一句口号：“网络就是计算机。” 根据他们多年的经验，太阳公司的许多科学家提出了以下常见的谬误列表：

1. 网络是可靠的。
2. 延迟为零。
3. 带宽是无限的。
4. 网络是安全的。
5. 拓扑结构不会改变。
6. 只有一个管理员。
7. 传输成本为零。
8. 网络是同质的。

### *谬误：* 网络是可靠的

贝利斯和金斯伯里合著的一篇名为《网络是可靠的》的论文（参见 [`https://queue.acm.org/detail.cfm?id=2655736`](https://queue.acm.org/detail.cfm?id=2655736)）审视了这一谬误。它发现了许多实例，例如微软报告其数据中心每天发生 5.2 次设备故障和 40.8 次链路故障。

中国政府使用“DNS 投毒”作为其审查其认为不良网站的技术之一。中国也运行着一个 DNS 根服务器。2010 年，该服务器配置错误，导致许多其他国家的 DNS 服务器被投毒。这使得许多非中国网站在中国境内外都无法访问（参见 [`http://www.pcworld.com/article/192658/article.html`](http://www.pcworld.com/article/192658/article.html)）。

还有许多其他可能的情况，例如 DDoS（分布式拒绝服务）攻击导致网站无法访问。在博克斯山学院，一名承包商曾用反铲挖土机挖断了连接我们 DHCP 服务器与网络其余部分的光纤电缆，结果我们当天剩下的时间就回家了。

网络*并不可靠*。这意味着任何联网程序都必须准备好应对故障。这导致了 Java 的 `RMI` 及大多数后续框架的设计选择，应用程序设计允许每次网络调用可能失败。



### 谬误：延迟为零

延迟是指发送信号到收到回复之间的时间差。在单进程系统中，延迟可能取决于函数调用返回前执行的计算量；而在网络上，延迟通常仅仅是因为数据需要穿越传输链路，并沿途被路由器等各种节点处理所致。

`ping`命令是展示延迟的好方法。从墨尔本向谷歌的澳大利亚服务器发送 ping 请求，大约需要 20 毫秒。向百度的中国服务器发送 ping 请求，则需要大约 200 毫秒。^(¹)

相比之下，Williams 的文章（参见[`http://www.eetimes.com/document.asp?doc_id=1200916`](http://www.eetimes.com/document.asp?doc_id=1200916)）讨论了 Linux 调度器的延迟，得出的平均延迟为 88 *微*秒。网络调用的延迟要高出数千倍。另一篇著名的文章是 P. Norvig 的“须知数字”，其中展示了延迟发生在何处以及各组件如何影响延迟（参见[`https://norvig.com/21-days.html#answers`](https://norvig.com/21-days.html#answers)）。

延迟会极大地影响分布式计算的设计和功能。例如，在计算调度决策时，如果数据收集速度过慢，就可能导致做出错误的决策。

### 谬误：带宽无限

任何在下载文件时去泡杯茶或咖啡的人都知道这是个谬误。我运行着自己的网络服务器，在 ADSL2 网络下上传速度为 800 Kbps。很不幸，我家用的是 HFC 网络，而灾难性的澳大利亚国家宽带网络要到 2020 年（三年后）才能将其升级到 1000 Kbps。

与此同时，我使用本地无线连接获得了 75 Mbps 的上传和下载速度，但这仍然不够快！

在 2021 年的修订版中，Ron 在旧金山通过电缆调制解调器获得了 400 Mbps 下载和 12 Mbps 上传速度。未来，我们会用 Starlink（或主流的卫星提供商）的数据进行更新；截至 2022 年，星链据称能提供 200 Mbps 下载和 20 Mbps 上传速度。

### 谬误：网络是安全的

科技公司强烈推动在所有网络通信中使用强加密，而世界各国政府也同等强烈地推动采用较弱系统或“仅供特定政府使用”的后门。这一点似乎对*民主*（我偶然拼写错误，但可能很准确！）政府和极权政府都同样适用。

当然，除此之外，还有一般的“坏蛋”，他们窃取和贩卖数以百万计的信用卡号和密码。

### 谬误：拓扑结构不变

事实并非如此。通常，这会影响延迟和带宽。但是，对路由或 IP 地址进行越多的硬编码，网络应用就越容易发生故障。如今，迁移到云端甚至云原生的情况并不少见，这正导致网络应用的集成方式发生许多变化。

### 谬误：只有一个管理员

那又怎样？当一切运转正常时没问题。问题出在出故障时——该怪谁？谁来修复？

多年来一个主要的研究课题是*网格计算*，它通常将计算任务分布到许多大学和研究机构，以解决庞大的科学问题。由于存在多个管理员、不同的访问和安全问题、不同的维护计划等等，这必须解决许多复杂的问题。云计算的出现*解决*了其中许多问题，减少了管理员和系统的数量，因此云计算比许多网格系统更具弹性。

### 谬误：传输成本为零

一旦我买了电脑，从 CPU 到显示器的传输成本是零（好吧，少量的电费！）。但我们都每月向网络提供商付费，因为他们需要建造服务器机房、铺设线缆等等。这是一个必须考虑的成本（例如，每月付费或被迫观看广告）。

### 谬误：网络是同质的

网络不是同质的，终端设备也不是——例如，你我的 PC、iPad、安卓设备和手机——更不用说物联网带来了无数联网设备。厂商不断地尝试进行产品锁定，而不断受限的工作环境也试图简化其控制系统，这些尝试在某种程度上是成功的。但当它们失败时，依赖同质性的系统也会失败。

当我们构建应用程序并学习相关技术时，记住我们之前讨论的分层模型是值得的。它将帮助我们分类目的和活动。客户端和服务器可能不是相同的硬件或软件，但我们可以对它们进行建模，这有助于我们理解并理清一切。

## 结论

本章试图强调，与其他计算风格相比，分布式计算有其独特的特性。忽视这些特性只会导致最终系统的失败。人们不断尝试简化架构模型，最近的是“微服务”和“无服务器”计算，但归根结底，复杂性依然存在。

这些问题必须使用任何编程语言来解决，后续章节将探讨 Go 语言如何应对它们。
